    Sensors Overview

    This section documents all sensor modules used in the CPS HHBK system for environmental data acquisition.

    What Are Sensors?

    In the context of cyber-physical systems, sensors are input devices that measure physical properties from the environment and convert them into digital signals that can be processed by the computer system.

    graph LR
    A[Physical World] -->|Heat, Light, etc.| B[Sensor]
    B -->|Analog Signal| C[Conversion]
    C -->|Digital Data| D[Software]

    Sensor Architecture

    Module Structure

    All sensor modules in the sensors/ directory follow a consistent structure:

    # sensors/<sensor_name>.py

def read_<data>_raw():
    """Low-level function to read raw data from hardware"""
    # Direct hardware access
    # Minimal processing
    return raw_data

def read_<data>():
    """High-level function with validation and conversion"""
    # Calls read_<data>_raw()
    # Validates data integrity
    # Converts to usable units
    return processed_data

    Design Principles

    1. Abstraction: Hide hardware-specific details
    2. Validation: Ensure data quality before returning
    3. Unit Conversion: Return data in standard units
    4. Error Handling: Gracefully handle sensor failures
    5. Documentation: Clear docstrings and comments

    Sensor Interfaces

    1-Wire Protocol (DS18B20)

    The KY-001 sensor uses the 1-Wire protocol, a serial communication protocol by Maxim Integrated (formerly Dallas Semiconductor).

    Characteristics:

    • Single data wire (plus ground)
    • Multiple devices on same bus
    • Parasitic power option available
    • Built-in addressing for device identification

    Implementation:

    On Raspberry Pi, 1-Wire is accessed through the Linux kernel driver via sysfs:

    /sys/bus/w1/devices/
   28-XXXXXXXXXXXX/
       w1_slave

    Data Format:

    f0 01 4b 46 7f ff 0c 10 5c : crc=5c YES
f0 01 4b 46 7f ff 0c 10 5c t=31000

    Line 1: CRC validation (YES = valid) Line 2: Temperature value (t=31000 means 31.000�C)

    GPIO Interface

    Future sensors might use direct GPIO:

    • Digital input (HIGH/LOW)
    • Pulse counting
    • PWM input
    • Interrupt-driven

    I2C Interface

    For advanced sensors (not currently used):

    • Two-wire communication (SDA, SCL)
    • Multiple devices with addressing
    • Higher data rates
    • Common for environmental sensors

    SPI Interface

    For high-speed sensors (not currently used):

    • Four-wire communication
    • Very fast data transfer
    • Common for IMUs, displays

    Sensor Data Flow

    flowchart TD
    A[Physical Phenomenon] -->|Temperature Change| B[DS18B20 Chip]
    B -->|1-Wire Protocol| C[Raspberry Pi GPIO 4]
    C -->|Linux Kernel Driver| D[/sys/bus/w1/devices/]
    D -->|File Read| E[read_temp_raw]
    E -->|Parse & Validate| F[read_temp]
    F -->|Return Data| G[Application Logic]

    Sensor Calibration

    Why Calibrate?

    Sensors may have offsets or drift over time. Calibration ensures accuracy.

    Calibration Methods

    Offset Calibration

    # Add calibration offset
TEMP_OFFSET = -0.5  # Sensor reads 0.5�C too high

def read_temp_calibrated():
    temp_c, temp_f = read_temp()
    temp_c += TEMP_OFFSET
    temp_f = temp_c * 9.0 / 5.0 + 32.0
    return temp_c, temp_f

    Two-Point Calibration

    For better accuracy across range:

    # Known reference points
REF_LOW = 0.0    # Ice water (0�C)
SENSOR_LOW = 0.2  # What sensor reads

REF_HIGH = 100.0  # Boiling water (100�C)
SENSOR_HIGH = 99.5 # What sensor reads

def calibrate(sensor_reading):
    """Apply two-point linear calibration"""
    slope = (REF_HIGH - REF_LOW) / (SENSOR_HIGH - SENSOR_LOW)
    offset = REF_LOW - (slope * SENSOR_LOW)
    return (sensor_reading * slope) + offset

    Sensor Testing

    Unit Tests

    # tests/test_sensors.py
import unittest
from sensors import ky001

class TestKY001(unittest.TestCase):

    def test_read_temp_returns_tuple(self):
        """Test that read_temp returns (celsius, fahrenheit)"""
        result = ky001.read_temp()
        self.assertIsInstance(result, tuple)
        self.assertEqual(len(result), 2)

    def test_temp_conversion(self):
        """Test Celsius to Fahrenheit conversion"""
        temp_c = 25.0
        temp_f = temp_c * 9.0 / 5.0 + 32.0
        self.assertAlmostEqual(temp_f, 77.0, places=1)

    def test_temp_in_valid_range(self):
        """Test temperature is within physically possible range"""
        temp_c, temp_f = ky001.read_temp()
        self.assertGreater(temp_c, -50)
        self.assertLess(temp_c, 150)

    Best Practices

    Do:

    • Validate all sensor data before use
    • Implement retry logic for transient failures
    • Log sensor errors for debugging
    • Use type hints for clarity
    • Document expected ranges
    • Handle missing sensors gracefully

    Don't:

    • Trust raw sensor data without validation
    • Ignore error conditions
    • Use infinite retry loops
    • Hard-code sensor paths or IDs
    • Skip data sanitization
    • Ignore outliers

    Performance Considerations

    Sensor Read Timing

    import time

# Measure sensor read time
start = time.time()
temp = read_temp()
duration = time.time() - start

print(f"Sensor read took {duration*1000:.1f}ms")
# Typical: 750-1000ms for DS18B20 at 12-bit resolution

    Optimization Strategies

    1. Lower Resolution: Faster reads, less precision
    2. Caching: Store recent values with TTL
    3. Async Reading: Non-blocking sensor access
    4. Batch Processing: Read multiple sensors concurrently
