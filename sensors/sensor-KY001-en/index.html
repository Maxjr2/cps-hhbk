    KY-001 Temperature Sensor (DS18B20)

    Overview

    This page documents the sensors/ky001.py Python module which reads temperature data from a KY-001 (DS18B20) 1-Wire temperature sensor and explains how to integrate it into main.py.

    Requirements

    • The Linux kernel must have the 1-Wire drivers loaded (for example w1-gpio and w1-therm).
    • The sensor device is exposed under /sys/bus/w1/devices/. Devices typically appear in a directory starting with 28-.
    • The module reads the w1_slave file in the device folder. The process needs read permission for that file.

    Wiring

    • Typical wiring for the KY-001 (DS18B20):
    • VCC -> 3.3V
    • GND -> GND
    • DATA -> 1-Wire data line (use a ~4.7kΩ pull-up resistor to 3.3V if not provided on your board)

    API (brief)

    The module exposes two functions:

    • read_temp_raw() -> list[str]

    • Reads the raw w1_slave file and returns its lines as a list of strings.

    • read_temp() -> (temp_c, temp_f) | None

    • Calls read_temp_raw() until the first line ends with YES, then parses the t= value from the second line and returns a tuple (temp_c, temp_f) where temp_c is Celsius and temp_f is Fahrenheit. If a device is not present the module may raise an error at import time (see notes).

    Example

    Basic usage:

    from sensors import ky001

result = ky001.read_temp()
if result is None:
  print("Temperature could not be read")
else:
  temp_c, temp_f = result
  print(f"Temperature: {temp_c:.2f} °C / {temp_f:.2f} °F")

    Integration with main.py

    The repository contains a main.py that imports sensors.ky001 and calls read_temp() to decide LED colors. Important notes about integration:

    • The current sensors.ky001.read_temp() returns a tuple (temp_c, temp_f). main.py must unpack this tuple and use the Celsius value (or Fahrenheit, if desired) for numeric comparisons.
    • The current main.py code calls temperature = sensors.ky001.read_temp() and then compares temperature < 21. That will raise a TypeError because temperature is a tuple. To fix this, unpack the result and handle the None case.

    Recommended, safe pattern for main.py:

    from sensors import ky001
import actors.led

def get_sensor_state():
  result = ky001.read_temp()
  if not result:
    # handle missing sensor or read error
    actors.led.set_led("yellow")
    return

  temp_c, temp_f = result
  # choose LED color based on Celsius
  if temp_c < 21:
    actors.led.set_led("green")
  elif temp_c >= 31:
    actors.led.set_led("red")
  elif temp_c >= 26:
    actors.led.set_led("yellow")


actors.led.selftest()

    Notes on the logic above:

    • Always check that read_temp() returned a value before unpacking. If read_temp() can block for a long time (waiting for raw data to report YES), consider calling it from a background thread or using a timeout mechanism.
    • The order of range checks matters: check the highest threshold first or use an explicit range mapping.

    Common issues and caveats

    • Import-time device lookup: the module currently resolves the device folder using glob.glob(base_dir + '28*')[0] at import time. If no 1-Wire device is present this will raise IndexError during import. Recommended alternatives:
    • Move device discovery into a helper function that runs on first read.

    • Wrap the initial device lookup in try/except and raise a clearer error.

    • Blocking reads: read_temp() polls the device file and sleeps (time.sleep(0.2)) until the data is marked valid. On missing or flaky hardware this can block the caller.

    • Permissions: ensure the process user has read access to /sys/bus/w1/devices/.../w1_slave.

    Suggested module improvements

    • Delay device-folder detection until the first reading.
    • Return a richer result object or raise custom exceptions for error cases instead of returning None.
    • Use with open(...) as f: when reading files to ensure proper resource handling.
